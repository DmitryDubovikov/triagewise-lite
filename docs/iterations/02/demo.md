# Демо / смок — Итерация 02 (promptfoo: CI eval-gate)

**Простыми словами, что тут построено.** Мы завели «экзамен» для промпта: 40 эталонных тикетов, по
которым проверяется, что промпт по-прежнему выдаёт правильно устроенные ответы. Экзамен встроен в
GitHub: каждый push и каждый PR прогоняют его автоматически, и если после чьей-то правки промпт
«экзамен» провалил — на PR загорается красный крестик и кнопка Merge блокируется. Причём прогон
экзамена бесплатный: он не обращается к OpenAI, а перечитывает ответы модели, записанные заранее.

**Что мы доказываем этим прогоном.** Демо проходит экзамен со всех сторон глазами скептика: где
лежит эталон и правда ли он версионируется; как проверка работает без сети и без ключа; и главное —
что она **действительно краснеет**, когда промпт ломают. Ворота, которые не умеют краснеть, — не
ворота. Ради этого итерация и затевалась: правка промпта, ломающая контракт выхода
(JSON по схеме `TriageResult`), больше не может доехать до main — это north-star строка
«CI eval-gate / regression testing для LLM».

Демо в двух частях:
- **Часть 1 — всё локально** (шаги 1–10): быстро, офлайн, $0; включая «сломать и увидеть красное».
- **Часть 2 — через GitHub** (шаги 11–12): пуш → убедиться, что зелёное; PR с поломкой → убедиться,
  что красное и мёрдж заблокирован.

Все команды копипаст-исполнимы из корня репозитория `/Users/dd/projects/pet/triagewise-lite`;
все шаги, кроме помеченных «⚠️ live», ничего не стоят. Поднятый MLflow этой итерации не нужен.

Перед началом:
```bash
cd /Users/dd/projects/pet/triagewise-lite
nvm use   # promptfoo 0.121.x требует Node >=22.22; версия зафиксирована в .nvmrc (22.23.1)
```

# Часть 1 — локально

## 1. Проект в принципе живой — статический гейт

**Простыми словами:** прежде чем показывать новое, убеждаемся, что база в порядке — код проходит
линтер, типы и обычные тесты. Если тут красное, дальше идти бессмысленно.

```bash
make check
```
**Ожидаемо:** все четыре шага зелёные —
```
uv run ruff check .            → All checks passed!
uv run ruff format --check .   → 26 files already formatted
uv run mypy app                → Success: no issues found in 16 source files
uv run pytest                  → 13 passed
```

## 2. Эталон хранится не «просто в папке» — проверяем DVC-хранилище круговоротом

**Простыми словами:** эталонные 40 тикетов лежат в файле `data/golden.jsonl`, но в git этот файл
**не** хранится. В git лежит только «квитанция» — крошечный файл `data/golden.jsonl.dvc` с
отпечатком содержимого (md5). Сам файл живёт в отдельном хранилище (папка рядом с репо), и DVC умеет
по квитанции его оттуда доставать. Проверяем это честно: удаляем файл, просим DVC вернуть его и
сверяем отпечаток возвращённого файла с квитанцией.

```bash
uv run dvc status -c
rm data/golden.jsonl
uv run dvc pull
md5 -q data/golden.jsonl
grep 'md5:' data/golden.jsonl.dvc
```
**Ожидаемо:** сначала `Cache and remote 'local' are in sync.` (всё, что мы клали в хранилище, там
есть); после удаления `dvc pull` печатает `1 file added` и файл возвращается; отпечатки совпадают:
```
7b60e7cdacfa47cb6841ff1c1d16407e
- md5: 7b60e7cdacfa47cb6841ff1c1d16407e
```
Повторный `uv run dvc pull` — no-op (`Everything is up to date.`).

## 3. Заглянуть в сам эталон

**Простыми словами:** посмотрим глазами, что внутри: 40 тикетов, у каждого проставлен «правильный
ответ» (категория, приоритет, тон, нужен ли человек). Восемь из них — нарочно каверзные: например,
вежливое по форме письмо, за которым на самом деле недовольный клиент. Каверзные заведены на вырост —
чтобы будущим, более строгим проверкам (iter 6) было на чём спотыкать плохие промпты.

```bash
wc -l data/golden.jsonl
jq -s 'map(select(.joker != null)) | group_by(.joker) | map({(.[0].joker): length}) | add' data/golden.jsonl
jq -c 'select(.joker == "hidden_negative")' data/golden.jsonl | head -1 | jq '{subject, expected, joker}'
```
**Ожидаемо:** `40`, раскладка каверзных `{"ambiguous_category": 3, "hidden_negative": 3,
"needs_human_edge": 2}` и пример скрытого негатива — вежливая тема письма с разметкой
`sentiment: negative`.

## 4. Из эталона генерятся файлы для проверки — и они не разъехались

**Простыми словами:** у GitHub нет доступа к нашему локальному DVC-хранилищу, поэтому проверка в CI
читает не сам эталон, а три файла, заранее сгенерированных из него и закоммиченных в git: конфиг
проверки, список из 40 тестов и текст промпта. Здесь мы запускаем генератор повторно и убеждаемся,
что он выдал байт-в-байт то же самое, что уже лежит в git, — значит, закоммиченные файлы не отстали
от эталона.

```bash
make eval-build
git status --porcelain eval/
```
**Ожидаемо:** генератор перечисляет три файла —
```
wrote eval/promptfooconfig.yaml
wrote eval/tests.yaml
wrote eval/prompts/champion.json
3 assets rebuilt into /Users/dd/projects/pet/triagewise-lite/eval
```
а `git status --porcelain eval/` не выводит **ничего**: файлы не изменились. В шапке
`eval/tests.yaml` виден тот же отпечаток эталона `7b60e7cd…`, что и в квитанции из шага 2, — по нему
рассинхрон был бы виден сразу.

## 5. Запустить саму проверку — зелёная, бесплатная, без ключа

**Простыми словами:** это тот самый экзамен, ровно той командой, которой его гоняет GitHub. Хитрость,
делающая его бесплатным: ответы модели на все 40 тикетов записаны заранее в файл `eval/outputs.json`
(в git), и проверка не звонит в OpenAI, а достаёт ответ из этого файла и валидирует его формат —
правильный ли JSON, все ли пять полей на месте, допустимые ли значения.

```bash
make eval
```
**Ожидаемо:** сводка promptfoo `✓ 40 passed (100%) / 0 failed (0%) / 0 errors (0%)` и финальная
строка:
```
eval gate green: 40 tests passed
```
Под капотом провайдер promptfoo подменён на наш файл-реплей (`--providers file://replay_provider.py`),
а последняя строка — вердикт jq-проверки: она требует «ошибок ноль, падений ноль и прогнались все
40», потому что сам promptfoo может выйти нулевым кодом, даже оборвав прогон.

## 6. Заглянуть в записанные ответы

**Простыми словами:** посмотрим на тот файл, из которого проверка берёт ответы: какой именно моделью
они записаны и как выглядит один живой ответ. Это настоящие ответы `gpt-4.1-nano`, записанные один
раз за ≈$0.01.

```bash
jq -r '.model' eval/outputs.json
jq '.outputs | length' eval/outputs.json
jq -r '.outputs | to_entries[0] | "key=\(.key) id=\(.value.id)\n\(.value.output)"' eval/outputs.json
```
**Ожидаемо:** `openai:chat:gpt-4.1-nano-2025-04-14` (датированный снапшот — модель запиннена), `40`,
и одна запись: ключ-отпечаток промпта, id тикета (`GS-009`) и JSON-ответ с пятью полями триажа.

## 7. Сломать и увидеть красное — локально ($0)

**Простыми словами:** теперь главное. Изображаем разработчика, который «улучшил» промпт, но забыл
перезаписать эталонные ответы: меняем текст промпта в коде, перегенерируем файлы проверки и запускаем
экзамен. Ответы в `eval/outputs.json` привязаны к старому тексту промпта (ключ — его отпечаток),
поэтому для нового текста ни одного ответа не найдётся — проверка обязана упасть. Потом возвращаем
всё как было.

```bash
sed -i '' 's/No prose, no code fences\./Reply in friendly prose, no JSON./' app/persistence/prompts.py
make eval-build
make eval; echo "make exit=$?"
```
**Ожидаемо:** таблица promptfoo полна строк
`[ERROR] no recorded output (key …) — prompt changed? re-record with `make eval-record` …`,
сводка `✗ 40 errors (100%)`, и `make exit=2` — не ноль, в CI это красный job. (promptfoo здесь сам
выходит не-нулём, оборвав прогон; jq-вердикт из шага 5 — страховка на случаи, когда он обрывается
«тихо», нулевым кодом.)

Возвращаем как было и убеждаемся, что снова зелёное и следов не осталось:

```bash
git restore app/persistence/prompts.py
make eval-build
make eval
git status --porcelain eval/ app/persistence/prompts.py
```
**Ожидаемо:** `eval gate green: 40 tests passed`, рабочая копия чистая.

## 8. Второй способ сломаться — плохой промпт, честно перезаписанный (⚠️ live, НЕ гоняем)

**Простыми словами:** шаг 7 ловил «поменял и не перезаписал». Есть и второй сценарий: разработчик
перезаписал ответы честно, но промпт стал настолько плох, что модель отвечает обычным текстом вместо
JSON — тогда ответы находятся, но проваливают проверку формата. Этот сценарий требует живых вызовов
модели (≈$0.01), поэтому здесь не гоняем — смотрим сохранённый лог того, как он был прогнан один раз
при закрытии итерации.

```bash
cat specs/02/red-demo.log
```
**Ожидаемо:** секция `2. assert-red (live)` — испорченный промпт («…Do not use JSON.»), пример
ответа-прозы, причина падения `Expected output to be valid JSON` и итог `✗ 40 failed (100%)`.

## 9. Автоматические стражи — чтобы файлы не разъехались без нас (вспомогательная проверка)

**Простыми словами:** вся конструкция держится на том, что несколько файлов согласованы между собой
(эталон ↔ сгенерированные файлы ↔ записанные ответы ↔ промпт в коде). Руками это в демо проверяли
шаги 4–7; ещё это стерегут обычные pytest-тесты — например, «ответы записаны той же моделью, что
указана в конфиге» и «файл промпта совпадает с промптом в коде».

```bash
uv run pytest tests/test_replay_provider.py tests/test_eval_assets.py -v
```
**Ожидаемо:** `8 passed` — 5 тестов реплея/артефактов (гоняются и в CI) и 3 теста эталона (локально
`PASSED`; в CI скипаются — эталона там нет по построению, см. шаг 4).

## 10. ⚠️ Перезаписать эталонные ответы (live, стоит денег) — НЕ гоняем

**Простыми словами:** когда промпт меняют по-настоящему (осознанно), эталонные ответы нужно записать
заново — иначе ворота останутся красными (шаг 7 это показал). Вот команда, которая это делает; она
звонит в OpenAI 40 раз, поэтому запускается только по явному решению.

```bash
# ⚠️ live, стоит денег (правило 4): 40 реальных вызовов gpt-4.1-nano (≈$0.01).
# Ключ выставить способом, который знаете только вы: положить OPENAI_API_KEY=sk-... в .env
# make eval-record
```
**Ожидаемо (если бы запускали):** promptfoo гоняет 40 тестов через настоящий OpenAI-провайдер, затем
экстрактор печатает `wrote /Users/dd/projects/pet/triagewise-lite/eval/outputs.json (40 recorded
outputs)` — обновлённый файл коммитится вместе с изменением промпта. Повторная перезапись без
изменения промпта бесплатна (локальный кэш promptfoo).

# Часть 2 — через GitHub

Здесь то же самое, но на настоящей поверхности: репозиторий
https://github.com/DmitryDubovikov/triagewise-lite, GitHub Actions гоняет оба job'а (`check` и
`eval-gate`) на каждый push/PR, а branch protection на `main` требует, чтобы оба прошли, — иначе
мёрдж заблокирован. Всё по-прежнему $0: секретов у CI нет, `eval-gate` только реплеит.

## 11. Запушить — и убедиться, что в GitHub зелёное

**Простыми словами:** отправляем текущий main на GitHub и смотрим, как GitHub сам, без нас, прогоняет
обе проверки. Это доказывает, что ворота живут не только на нашей машине.

```bash
git push
gh run watch $(gh run list --branch main --limit 1 --json databaseId --jq '.[0].databaseId') --exit-status
```
**Ожидаемо:** прогон завершается успешно, оба job'а с галочками:
```
✓ main ci · <run-id>
JOBS
✓ check in ~1m
✓ eval-gate in ~1m
```
В логе `eval-gate` видно `Evaluation completed: 40/40 tests` — тот же экзамен, что в шаге 5, только
на раннере GitHub. (Если пушить нечего — команда `gh run watch …` просто покажет последний прогон.)

## 12. Сломать в ветке, открыть PR — и убедиться, что красное и мёрдж заблокирован

**Простыми словами:** финальная проверка всей затеи. Делаем ту же вредную правку, что в шаге 7, но
как сделал бы коллега: в отдельной ветке, с push и pull request'ом. GitHub прогоняет проверки, они
краснеют, и кнопка Merge гаснет — испорченный промпт физически не может попасть в main. Потом
закрываем PR и удаляем ветку.

```bash
git switch -c demo/red-eval-gate
sed -i '' 's/No prose, no code fences\./Reply in friendly prose, no JSON./' app/persistence/prompts.py
make eval-build
git add -A && git commit -m "demo: вредная правка промпта без перезаписи outputs.json"
git push -u origin demo/red-eval-gate
gh pr create --title "demo: harmful prompt change — eval-gate must go red" --body "Демо-PR, будет закрыт без мёржа."
gh pr checks --watch
gh pr view --json mergeStateStatus --jq .mergeStateStatus
```
**Ожидаемо:** `gh pr checks --watch` дожидается прогона, оба job'а красные:
```
check      fail  …/actions/runs/…
eval-gate  fail  …/actions/runs/…
```
`eval-gate` падает по замыслу итерации — в его логе те же `[ERROR] no recorded output (key …)`, что в
шаге 7. `check` краснеет за компанию, и это честно: записанный офлайн-ответ для обычного smoke-теста
(кассета) тоже привязан к тексту промпта, поэтому `test_triage_champion_replay_smoke` падает на
«No cassette for tier 'cheap'» — регрессию промпта ловят оба гейта. Главное: `mergeStateStatus`
печатает `BLOCKED` — required checks не пускают мёрдж. (Сценарий прогнан вживую — PR #1, закрыт без
мёржа.)

Прибираем за собой — закрываем PR, удаляем ветку локально и на GitHub, возвращаемся на `main`:

```bash
gh pr close demo/red-eval-gate --comment "демо отработало: eval-gate красный, merge был BLOCKED" --delete-branch
```
**Ожидаемо:** `✓ Closed pull request …`, `✓ Deleted branch demo/red-eval-gate`, шелл снова на `main`,
рабочая копия чистая.
