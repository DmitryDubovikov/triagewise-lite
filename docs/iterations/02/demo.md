# Демо / смок — Итерация 02 (promptfoo: CI eval-gate)

**Что мы доказываем этим прогоном.** Итерация поставила регрессионные ворота перед промптом: эталонный
golden-сет из 40 размеченных тикетов версионируется DVC, promptfoo прогоняет по нему champion-промпт и
ассертит формат каждого ответа (валидный JSON по схеме `TriageResult`), а в GitHub Actions это
отдельный job `eval-gate` — офлайн, $0 и без единого секрета, потому что ответы модели записаны
заранее в закоммиченный реплей-артефакт `eval/outputs.json`. Смок проходит путь скептика: убеждаемся,
что golden реально живёт в DVC (round-trip через remote, не «файл лежит рядом»), что производные
ассеты генерятся из него детерминированно, что ворота зелёные на честном прогоне — и, главное, что они
**действительно краснеют**, когда промпт меняют без перезаписи артефакта. Ворота, которые не умеют
краснеть, — не ворота.

**Зачем это за пределами демо.** С этого момента правка промпта, ломающая контракт выхода, не может
доехать до main: CI заблокирует мёрдж. Это и есть north-star строка «CI eval-gate / regression testing
для LLM» — фундамент, на который iter 6 поставит сравнение champion-vs-challenger.

Все команды копипаст-исполнимы из корня репозитория `/Users/dd/projects/pet/triagewise-lite`;
все шаги, кроме помеченных «⚠️ live», ничего не стоят. Поднятый MLflow этой итерации не нужен —
ворота работают целиком офлайн.

Перед началом:
```bash
cd /Users/dd/projects/pet/triagewise-lite
nvm use   # promptfoo 0.121.x требует Node >=22.22; версия зафиксирована в .nvmrc (22.23.1)
```

## 1. Сначала убедиться, что код вообще зелёный — статический гейт

Прежде чем демонстрировать ворота, проверяем, что проект проходит линт/типы/тесты — иначе всё
дальнейшее бессмысленно. Заодно здесь гоняются оба новых тест-файла итерации (golden присутствует
локально, поэтому skipif-тесты реально выполняются, а не скипаются).

```bash
make check
```
**Ожидаемо:** все четыре шага зелёные —
```
uv run ruff check .            → All checks passed!
uv run ruff format --check .   → 26 files already formatted
uv run mypy app                → Success: no issues found in 16 source files
uv run pytest                  → 13 passed
```

## 2. Golden-сет действительно под DVC — round-trip через remote

Заявку «golden версионируется DVC» проверяем не взглядом на файл, а честным круговоротом: убеждаемся,
что кэш и remote синхронны (`dvc push` прошёл), удаляем рабочую копию и восстанавливаем её
`dvc pull` — затем сверяем md5 с указателем `data/golden.jsonl.dvc`, закоммиченным в git.

```bash
uv run dvc status -c
rm data/golden.jsonl
uv run dvc pull
md5 -q data/golden.jsonl
grep 'md5:' data/golden.jsonl.dvc
```
**Ожидаемо:** `Cache and remote 'local' are in sync.`; pull печатает `1 file fetched` (или added) и
файл возвращается; хэши совпадают:
```
7b60e7cdacfa47cb6841ff1c1d16407e
- md5: 7b60e7cdacfa47cb6841ff1c1d16407e
```
Повторный `uv run dvc pull` — no-op (`Everything is up to date.`): операция идемпотентна.

## 3. Посмотреть сам эталон — что именно лежит в golden

Артефакт стоит увидеть глазами, а не только его md5: 40 размеченных тикетов, из них 8 «джокеров» —
заведомо каверзных (двусмысленная категория, скрытый негатив, пограничный needs_human), ради которых
будущим воротам champion-vs-challenger (iter 6) будет что различать.

```bash
wc -l data/golden.jsonl
jq -s 'map(select(.joker != null)) | group_by(.joker) | map({(.[0].joker): length}) | add' data/golden.jsonl
jq -c 'select(.joker == "hidden_negative")' data/golden.jsonl | head -1 | jq '{subject, expected, joker}'
```
**Ожидаемо:** `40`, раскладка джокеров `{"ambiguous_category": 3, "hidden_negative": 3,
"needs_human_edge": 2}` и пример скрытого негатива — вежливый текст с разметкой `sentiment: negative`.

## 4. Производные ассеты генерятся из golden детерминированно

CI не умеет `dvc pull` (remote локальный), поэтому ворота работают по закоммиченным ассетам,
сгенерированным из golden. Проверяем, что генератор воспроизводит их байт-в-байт — рабочая копия
после прогона остаётся чистой, значит закоммиченное синхронно с источником (это же держит тест
`test_committed_assets_in_sync_with_golden`).

```bash
make eval-build
git status --porcelain eval/
```
**Ожидаемо:** скрипт перечисляет три файла и итог —
```
wrote eval/promptfooconfig.yaml
wrote eval/tests.yaml
wrote eval/prompts/champion.json
3 assets rebuilt into /Users/dd/projects/pet/triagewise-lite/eval
```
а `git status --porcelain eval/` не выводит **ничего** (те же байты — идемпотентность). В шапке
`eval/tests.yaml` виден md5 golden-сета — тот же `7b60e7cd…`, что в шаге 2.

## 5. Ворота зелёные: promptfoo реплеит записанные ответы — $0, без ключа

Это сам eval-gate, ровно та команда, которую CI гоняет на каждый push/PR. promptfoo прогоняет все 40
тестов, но провайдер подменён на файл-реплей (`--providers file://replay_provider.py`): ответы берутся
из закоммиченного `eval/outputs.json` по хэшу отрендеренного промпта, сеть не трогается, ключ не
нужен. Настоящие ворота — jq-пост-чек в Makefile: promptfoo выходит с 0 даже при аборте на ошибках
провайдера, поэтому зелёный засчитывается только если прогнались все 40 и ни ошибок, ни падений.

```bash
make eval
```
**Ожидаемо:** сводка promptfoo `✓ 40 passed (100%) / 0 failed (0%) / 0 errors (0%)` и финальная
строка пост-чека:
```
eval gate green: 40 tests passed
```

## 6. Посмотреть сам реплей-артефакт — что именно реплеится

Показываем содержимое `eval/outputs.json`: какой моделью записано (датированный снапшот — пин-гейт
держится и тут), сколько ответов и как выглядит один из них — настоящий JSON-ответ nano, записанный
на `make eval-record`.

```bash
jq -r '.model' eval/outputs.json
jq '.outputs | length' eval/outputs.json
jq -r '.outputs | to_entries[0] | "key=\(.key) id=\(.value.id)\n\(.value.output)"' eval/outputs.json
```
**Ожидаемо:** `openai:chat:gpt-4.1-nano-2025-04-14`, `40`, и одна запись — ключ-хэш, id тикета
(`GS-009`) и ответ модели: JSON с пятью полями триажа.

## 7. Ворота краснеют: правим промпт «во вред» без перезаписи — CI бы заблокировал мёрдж ($0)

Ключевая проверка итерации — регрессионный сценарий целиком, как его прожил бы разработчик: меняем
champion-шаблон в коде, перегенерируем ассеты (это сделал бы он сам — иначе покраснеет sync-тест) и
запускаем ворота, «забыв» перезаписать реплей-артефакт. Хэш отрендеренного промпта изменился → все 40
реплей-ключей промахиваются → провайдер отвечает ошибками → jq-пост-чек красный. Именно это увидел бы
PR в GitHub Actions.

```bash
sed -i '' 's/No prose, no code fences\./Reply in friendly prose, no JSON./' app/persistence/prompts.py
make eval-build
make eval; echo "make exit=$?"
```
**Ожидаемо:** таблица promptfoo полна строк
`[ERROR] no recorded output (key …) — prompt changed? re-record with `make eval-record` …`,
сводка `✗ 40 errors (100%)`, и `make exit=2` (не ноль — job в CI красный, мёрдж заблокирован).
В этом сценарии promptfoo абортит прогон и сам выходит не-нулём (100), так что make падает уже на
шаге promptfoo, не дойдя до jq-пост-чека; пост-чек — страховка на случаи, когда promptfoo выходит
нулём, не прогнав все тесты.

Возвращаем всё на место и убеждаемся, что ворота снова зелёные:

```bash
git restore app/persistence/prompts.py
make eval-build
make eval
git status --porcelain eval/ app/persistence/prompts.py
```
**Ожидаемо:** `eval gate green: 40 tests passed`, рабочая копия чистая — демо не оставило следов.

## 8. Второй путь к красному — испорченный промпт ломает формат-ассерты (⚠️ live, в церемонии НЕ гоняем)

Miss-red (шаг 7) ловит «поменял и не перезаписал». Второй сценарий — «перезаписал, но промпт стал
хуже»: если champion-промпт испортить так, что модель отвечает прозой вместо JSON, и честно
перезаписать артефакт live-прогоном, то реплей попадает в ключи, но `is-json`-ассерты валят все 40
тестов. Этот путь требует живых вызовов (≈$0.01), поэтому в смоке не выполняется — он был прогнан
один раз при закрытии итерации, лог-доказательство закоммичен:

```bash
cat specs/02/red-demo.log
```
**Ожидаемо:** секция `2. assert-red (live)` — испорченный системный промпт («…Do not use JSON.»),
пример ответа-прозы, причина падения `Expected output to be valid JSON` и итог
`✗ 40 failed (100%)`, `exit=100`.

## 9. Стражи консистентности — то, что держит ворота честными в CI (вспомогательная проверка)

Основной путь — шаги 4–7 через продуктовую поверхность (`make eval`). Эти unit-тесты дублируют
гарантии там, где живого прогона нет: ключи реплея стабильны, промах громкий, записанная модель
совпадает с провайдером в конфиге (bump тира не останется зелёным на ответах старой модели),
закоммиченный файл промпта совпадает с шаблоном в коде, а ассеты — с golden.

```bash
uv run pytest tests/test_replay_provider.py tests/test_eval_assets.py -v
```
**Ожидаемо:** `8 passed` — 5 тестов провайдера/артефактов (они гоняются и в CI) и 3 golden-теста
(локально `PASSED`; в CI их скипает `skipif` — golden там отсутствует по построению).

## 10. ⚠️ Перезаписать реплей-артефакт (live, стоит денег) — в церемонии НЕ выполняем

Ради полноты — откуда берётся `eval/outputs.json`. Запускается только по явному go, когда промпт или
golden реально изменились (шаг 7 показывает, что без этого ворота красные):

```bash
# ⚠️ live, стоит денег (правило 4): 40 реальных вызовов gpt-4.1-nano через promptfoo (≈$0.01).
# Ключ выставить способом, который знаете только вы: положить OPENAI_API_KEY=sk-... в .env
# make eval-record
```
**Ожидаемо (если бы запускали):** promptfoo гоняет 40 тестов через родной OpenAI-провайдер, затем
`scripts/extract_eval_outputs.py` пишет `wrote /Users/dd/projects/pet/triagewise-lite/eval/outputs.json
(40 recorded outputs)` — файл коммитится вместе с изменением промпта. Повторная перезапись без
изменения промпта бесплатна (локальный кэш promptfoo в `eval/.promptfoo-cache/`, gitignored). В рамках
смок-церемонии шаг **пропущен** как live.
