# Решения и грабли — Итерация 02 (promptfoo: CI eval-gate)

## Решения и почему

- **CI = $0 через собственный файл-реплей, а не через закоммиченный кэш promptfoo.** Первоначальный
  план был соблазнительно прост: прогнать promptfoo live локально, закоммитить его кэш и в CI
  реплеить из кэша. План умер на факте: promptfoo хэширует в кэш-ключ **весь запрос, включая
  API-ключ** — без настоящего ключа в CI ни один закоммиченный кэш-хит не совпадёт. Итоговая схема:
  live-запись остаётся за родным OpenAI-провайдером promptfoo (`make eval-record`, гейтится, ≈$0.01),
  а `scripts/extract_eval_outputs.py` дистиллирует результаты в закоммиченный `eval/outputs.json`,
  где ключ — хэш отрендеренного промпта (без ключа и модели в ключе); CI реплеит его через
  собственный python-провайдер `eval/replay_provider.py` с оверрайдом `--providers`. Бонус против
  исходного плана: смена промпта инвалидирует все ключи разом, и промах — это громкая ошибка, а не
  тихий live-вызов. Развилка tech-decisions «promptfoo владеет своими вызовами» при этом цела — мы
  не встраивали его в `route()`, только подменили провайдера на реплей.

- **Настоящие ворота — jq-пост-чек, а не exit-код promptfoo.** Обнаружилось, что promptfoo выходит
  с кодом 0, когда *абортит* прогон на ошибках провайдера — то есть miss-red был бы «зелёным» для
  make. Поэтому цель `eval` пишет результаты в `eval/.results.json` и сама выносит вердикт через jq:
  зелёный только при `failures == 0 && errors == 0 && successes == <число записей в outputs.json>`.
  Последнее условие важно отдельно: «ошибок нет» ещё не значит «прогналось всё» — сверяем число
  успехов с размером артефакта.

- **Ассерты — только формат/поля (осознанный срез).** Единственный ассерт — `is-json` со схемой,
  сгенерированной из Pydantic-модели `TriageResult` (5 обязательных полей, enum'ы priority/sentiment,
  boolean needs_human). Разметка golden (правильная категория и т.д.) кладётся в `eval/tests.yaml`
  метаданными, но ни один ассерт по ней не судит: accuracy — не ворота (правило 1), label-гейт и
  сравнение champion-vs-challenger — территория iter 6. Джокеры (8 из 40) сейчас «спят» — их смысл
  проявится, когда гейту будет что различать.

- **Golden не в git, а CI работает по закоммиченным производным.** DVC-remote — локальная папка
  (`../triagewise-lite-dvc-remote`), CI сделать `dvc pull` не может по построению. Вместо этого
  `make eval-build` детерминированно генерит promptfoo-ассеты из golden, и они коммитятся. Риск
  рассинхрона источника и производных принят осознанно (golden почти статичен) и обложен с двух
  сторон: шапка каждого генерённого YAML несёт md5 golden-сета (тот же хэш, что в
  `data/golden.jsonl.dvc` — рассинхрон виден на глаз), а `test_committed_assets_in_sync_with_golden`
  сверяет байты локально. В CI этот тест скипается (golden там нет) — это принятая цена схемы.

- **Тест-файлы разрезаны по признаку «нужен ли golden».** `tests/test_eval_assets.py` целиком под
  `skipif` (без golden — скип, как в CI); всё, что обязано гоняться в CI, вынесено в
  `tests/test_replay_provider.py` — свойства реплей-провайдера плюс два стража консистентности
  закоммиченного: записанная модель == провайдер в конфиге (иначе bump тира в `llm-tiers.yaml`
  остался бы зелёным, реплея ответы старой модели), закоммиченный файл промпта == шаблон в коде
  (иначе правка `TRIAGE_CHAMPION_TEMPLATE` без `make eval-build` судила бы устаревший промпт).

- **Repo-tree-тесты читают пути из `Settings.model_construct()`, а не из `Settings()`.** Эти тесты
  проверяют закоммиченные файлы репозитория, поэтому обязаны быть глухи к окружению: транзиентный
  `TRIAGE_TIER=mid` в шелле разработчика не должен ни уронить, ни молча перенацелить проверку.
  `model_construct` отдаёт чистые дефолты полей — без чтения `.env` и env-переменных.

## Грабли

- **Кэш promptfoo солит ключ API-ключом** — главная грабля итерации, похоронившая первоначальный
  план «закоммитить кэш» (подробности в первом решении выше). Мораль: прежде чем строить CI-схему на
  чужом кэше, проверь, из чего сделан его ключ. Локально кэш всё же используется — но только как
  ускоритель повторной записи (`eval/.promptfoo-cache/`, gitignored), не как реплей-артефакт.

- **promptfoo запускает python-провайдер системным `python3`, вне uv-окружения.** Поэтому
  `eval/replay_provider.py` — нарочно stdlib-only и не импортирует `app/*`: у системного питона нет
  ни pydantic, ни нашего пакета. Мостик между мирами — `load_replay_provider()` в
  `app/workflow/eval_assets.py`, который импортирует модуль по пути файла: экстрактор и тесты
  используют ровно ту же функцию `prompt_key`, что и реплей, — запись и реплей не могут разъехаться.

- **Ключ реплея пришлось канонизировать.** Chat-промпт promptfoo рендерит в JSON-строку, и байтовое
  представление одного и того же промпта на записи и на реплее не обязано совпадать (пробелы, порядок
  ключей). `prompt_key` поэтому сначала парсит строку как JSON и пересобирает канонично
  (`sort_keys=True`, без пробелов), и только потом хэширует; не-JSON падает обратно на сырую строку.
  Свойство закреплено тестом `test_prompt_key_is_whitespace_and_order_insensitive`.

- **promptfoo проверяет версию Node-рантайма и жёстко отказывает.** Смок-церемония закрытия
  вскрыла: в свежем шелле (nvm-default = v22.17.0) `make eval` падает ещё до старта — promptfoo
  0.121.17 требует `^20.20.0 || >=22.22.0` (во время итерации прогоны шли из шелла с активным
  v22.23.1, поэтому проблема не всплывала). Починка: версия Node зафиксирована в `.nvmrc`
  (`22.23.1`), демо начинается с `nvm use`, а CI читает версию оттуда же
  (`node-version-file: .nvmrc` вместо плавающего `node-version: "22"`) — рантайм ворот больше не
  зависит от того, какой node случился в шелле.

- **Экстрактор отказывается писать частичный артефакт.** Если у какого-то теста в live-прогоне нет
  ответа (упавший вызов), `extract_eval_outputs` завершается ошибкой, а не пишет outputs.json без
  этой записи: частичный артефакт дал бы вечно-красный реплей с загадочным miss вместо внятного
  «почини live-прогон».

## Осознанные срезы и отложенные TODO

- **DVC-remote — локальная папка, не S3/облако** (`# dl-lite`: локальный dir-remote → облачный
  remote, если проект дорастёт до второй машины). Следствие принято: CI не тянет golden и работает
  по производным ассетам; golden-тесты в CI скипаются.

- **Красное CI-демо на самом GitHub — прогнано после закрытия итерации.** На close ворота были
  проверены локально обоими путями (miss-red $0 и assert-red live, evidence в
  `specs/02/red-demo.log`), а репозитория на GitHub ещё не существовало. Сразу после закрытия репо
  запушен (https://github.com/DmitryDubovikov/triagewise-lite), на `main` включён branch protection
  с required check'ами `check` + `eval-gate`, и красный сценарий прогнан вживую: PR #1 с вредной
  правкой промпта дал красный `eval-gate` (реплей-miss) и `mergeStateStatus: BLOCKED` — мёрдж
  заблокирован (демо, часть 2, шаг 12). Попутный факт: `check` на таком PR краснеет тоже — кассета офлайн-смока
  ключуется промптом, так что регрессию промпта ловят оба гейта.

- **Label/accuracy-ассерты и сравнение champion-vs-challenger — iter 6.** Разметка и джокеры уже в
  golden и уже едут в `eval/tests.yaml` метаданными — гейту iter 6 останется начать по ним судить.

- **Cost/latency не считаются (iter 3), дрейф не ловится (iter 5), Phoenix нет (iter 5).**
