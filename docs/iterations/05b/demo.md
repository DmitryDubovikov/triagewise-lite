# Демо итерации 05b — судья оценивает триаж на трафике, оценки видны в Phoenix

Этот прогон доказывает практику **online evaluation / LLM-as-judge в проде**: уже обслуженный трафик триажа выборочно перечитывает более сильная модель и ставит каждому разбору вердикт correct/incorrect прямо рядом с его трейсом — так саппорт-лид видит тихую деградацию качества, которую дрейф-монитор 5a по построению не ловит (распределение категорий может быть стабильным, а ответы — неправильными). Проверяем контур целиком: трафик затрейсен → судья сэмплирует и судит (⚠️ единственный принципиально live-шаг проекта) → стор Phoenix подтверждает оценки → повторный запуск судьи ничего не пере-оплачивает.

Все команды выполняются из корня репо `/Users/dd/projects/pet/triagewise-lite`.

### 1. Поднять control plane и убедиться, что трафик затрейсен

Судье нужен материал: спаны `triage_ticket` в Phoenix, которые оставляет `make traffic` (итерация 5a, replay, $0). Без них judge честно скажет, что судить нечего.

```bash
make up
uv run python -m scripts.drift_report >/dev/null 2>&1 && echo "spans on the store: OK" || make traffic
```

**Ожидаемо:** оба контейнера подняты; строка `spans on the store: OK` (drift-report видит спаны), либо — на свежем сторе — вывод `make traffic` с `11 tickets triaged … 10 tickets triaged`.

### 2. Судья ещё не запускался — отчёт честно красный

Это негативная половина done-gate: отчёт выносит вердикт по стору, и пока аннотаций нет, он обязан отличать «нечего показать» от «всё хорошо».

```bash
make judge-report; echo "exit=$?"
```

**Ожидаемо** (только на сторе, где судья ещё не бегал; после шага 3 этот шаг уже не воспроизвести — аннотации персистентны):

```
No 'triage_quality' annotations yet — run `make judge` (live, rule 4).
exit=1
```

### 3. Запустить судью ⚠️ live, стоит денег

Это и есть online LLM-as-judge: сэмпл трафика (детерминированная половина, `JUDGE_SAMPLE_RATE=0.5`) уходит на `smart`-тир (`gpt-4.1-2025-04-14` из `llm-tiers.yaml`), каждый вердикт приклеивается аннотацией к исходному спану. Цена: ~$0.002 на спан; на свежем сторе с одной парой пачек это 14 спанов ≈ $0.03 (важно: судятся *спаны*, не тикеты — каждый повторный `make traffic` перед этим шагом умножает число судимых дублей, наш прогон при 84 спанах стоил ≈ $0.11). Ключ нужен один раз: `echo 'OPENAI_API_KEY=sk-…' >> .env` (значение — ваш личный ключ OpenAI).

```bash
make judge
```

**Ожидаемо:** по строке `[span_id] label: объяснение` на каждый осуждённый спан и итог; в вердиктах есть и `incorrect` — судья ловит «джокера» golden-сета (позитивный отзыв о dark mode, который триаж записал в `feature_request`):

```
[331600921ea02d53] incorrect: The ticket is purely positive feedback about the new dark mode
and does not contain a feature request. ...
[b8321b65ae6d5982] correct: The category 'billing' fits since the issue is about a double charge. ...
...
56 spans judged at tier=smart (84 traced, 0 already judged, sample_rate=0.5)
```

(Числа `56/84` — от нашего стора с четырьмя прогонами трафика; на свежем сторе будет `14 spans judged … 21 traced`.)

### 4. Verify the store, не UI: оценки лежат в Phoenix

Ключевой шаг done-gate (правило 8): отчёт спрашивает сам стор Phoenix REST-клиентом и печатает, что судья реально записал.

```bash
make judge-report; echo "exit=$?"
```

**Ожидаемо:** JSON со счётчиками меток и повердиктным списком по тикетам, нулевой exit-код; `incorrect` сидит на тикете-джокере DW-007:

```json
{
  "annotation": "triage_quality",
  "spans_traced": 84,
  "spans_judged": 56,
  "labels": {
    "correct": 52,
    "incorrect": 4
  },
  "verdicts": [
    {"ticket_id": "DW-001", "label": "correct", "score": 1.0},
    ...
    {"ticket_id": "DW-007", "label": "incorrect", "score": 0.0},
    ...
  ]
}
exit=0
```

### 5. Идемпотентность: повторный судья — no-op и $0

Инкрементальность — часть done-gate: уже осуждённые спаны выбывают до вызова модели, поэтому второй запуск подряд не делает ни одного LLM-вызова (команда формально live, но на полностью осуждённом сторе тратит $0 по построению).

```bash
make judge 2>/dev/null | tail -1
```

**Ожидаемо:**

```
0 spans judged at tier=smart (84 traced, 56 already judged, sample_rate=0.5)
```

### 6. Посмотреть оценку рядом с трейсом глазами (вспомогательная проверка, не основной путь)

Стор уже всё подтвердил на шаге 4; UI — витрина тех же данных и источник скриншота «оценка рядом с трейсом».

Открыть http://localhost:6006 → проект `triagewise` → Traces → кликнуть любой спан `triage_ticket` из осуждённых.

**Ожидаемо:** на панели спана — блок аннотаций с `triage_quality`: label (`correct`/`incorrect`), score и explanation судьи; в списке трейсов у осуждённых спанов метка видна прямо в таблице.
