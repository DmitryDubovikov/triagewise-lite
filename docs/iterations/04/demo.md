# Демо / смок — Итерация 04 (semantic cache поверх access-layer)

**Простыми словами, что тут построено.** У конвейера появилась память на смысл: разобрал тикет —
близкие по формулировке собратья отдаются из кэша мгновенно и бесплатно, без похода в сеть и даже
без кассеты. Похожесть считает локальная embedding-модель (fastembed, $0 за вызов), исход каждого
вызова (`hit`/`miss`/`off`) пишется в SLO-журнал, а `make cache-stats` превращает журнал в цифру
hit-rate.

**Что мы доказываем этим прогоном.** Три вещи, глазами скептика. Первое: кэш действительно
*семантический* — попадание случается на тикете-парафразе DW-011, у которого **нет кассеты**, то
есть ответ мог прийти только из кэша (и контрольный прогон без кэша это подтверждает падением).
Второе: экономия видна цифрой — hit-rate считается из журнала, а не декларируется. Третье: кэш —
опция без побочных эффектов: выключен по умолчанию, повторные прогоны не распухают стор. Вместе это
закрывает вторую половину north-star строки «Productionized routing + semantic caching» (роутинг
дала iter 3).

Все команды копипаст-исполнимы из корня репозитория `/Users/dd/projects/pet/triagewise-lite`; все
шаги бесплатны — live-вызовов в этом демо нет вообще (единственная сеть — одноразовая загрузка
embedding-модели ~30 МБ на шаге 3, это не LLM-деньги). Для шагов 3-5 нужен поднятый MLflow (промпт
триажа живёт в реестре с iter 1) — шаг 2 это обеспечивает.

Перед началом:
```bash
cd /Users/dd/projects/pet/triagewise-lite
```

## 2. Поднять реестр и начать с чистого кэша

**Простыми словами:** триаж берёт промпт из MLflow-реестра, поэтому реестр должен быть поднят
(регистрация идемпотентна). Стор кэша и SLO-журнал сносим, чтобы прогон был детерминированным:
демо доказывает «miss → hit», а hit-rate шага 7 должен считаться ровно по этому прогону (оба
файла — гитигноренные runtime-артефакты, их потеря ничего не ломает).

```bash
make up
uv run python -m scripts.register_prompt
rm -f logs/semantic_cache.jsonl logs/llm_calls.jsonl
```
**Ожидаемо:** контейнер `mlflow` поднят (localhost:5050); скрипт отчитывается, что
champion/challenger на месте, без создания лишних версий при повторе.

## 3. Разобрать DW-001 при включённом кэше — первый прогон, честный промах

**Простыми словами:** кэш включается одной переменной окружения — как переключение тиров. Память
пуста, поэтому тикет идёт обычным путём (replay-кассета), но по дороге кэш запоминает пару
«текст-вектор → ответ». В SLO-строке появился новый хвост `cache=miss`.

```bash
SEMANTIC_CACHE_ENABLED=1 uv run python -m app.cli.main DW-001
```
**Ожидаемо:** при самом первом запуске fastembed один раз скачает модель `bge-small-en-v1.5`
(~30 МБ, прогресс-бары в stderr; из-за этого latency первого вызова — секунды). Дальше — SLO-след
с `cache=miss` и обычный JSON триажа:
```
[slo] cheap->gpt-4.1-nano-2025-04-14 (replay) …ms $0.000038 (cassette) cache=miss ok
[DW-001] cheap (replay)
{ "category": "Login Issues", "priority": "high", "sentiment": "negative", ... }
```

## 4. Разобрать парафраз DW-011 — семантическое попадание без кассеты

**Простыми словами:** это главный аргумент итерации. DW-011 — тот же смысл, что DW-001 («не могу
войти после сброса пароля»), но другими словами, и **кассеты для него не существует**. В replay без
кэша ему просто неоткуда взять ответ — а с кэшем он обслуживается попаданием: косинус к DW-001
выше порога 0.90. Сначала попадание, потом контрольный прогон без кэша — чтобы исключить фокусы.

```bash
SEMANTIC_CACHE_ENABLED=1 uv run python -m app.cli.main DW-011
uv run python -m app.cli.main DW-011
```
**Ожидаемо:** первая команда — попадание: `cache=hit`, цена $0.000000, источник `none` (не
кассета!), и полный JSON триажа — тот же ответ, что у DW-001. Вторая команда (кэш выключен, это
дефолт) — громкий `FileNotFoundError: No cassette for tier 'cheap' …` — доказательство, что ответ
DW-011 мог прийти только из семантического кэша:
```
[slo] cheap->gpt-4.1-nano-2025-04-14 (replay) 151ms $0.000000 (none) cache=hit ok
[DW-011] cheap (replay)
{ "category": "Login Issues", ... }
```

## 5. Повторить попадание — стор не растёт (идемпотентность)

**Простыми словами:** попадания не пополняют память — только промахи. Значит, повторные прогоны
сходятся к стабильному состоянию: одна запись на уникальный запрос, стор не распухает.

```bash
SEMANTIC_CACHE_ENABLED=1 uv run python -m app.cli.main DW-011 2>/dev/null >/dev/null
wc -l < logs/semantic_cache.jsonl
```
**Ожидаемо:** ровно `1` — после трёх кэшируемых вызовов (miss DW-001 + два hit DW-011) в сторе
одна-единственная запись.

## 6. Заглянуть в сам артефакт — что именно легло в стор

**Простыми словами:** показываем содержимое памяти, а не только её отражение в логе. Каждая запись
несёт неймспейс-перегородку (hash от LLM-модели + эмбеддера + системного промпта), исходный текст
(человекочитаемый след для отладки), 384-мерный вектор и сам ответ.

```bash
jq -c '{namespace: .namespace[0:12], text: .text[0:48], dims: (.vector | length), content: .content[0:40]}' logs/semantic_cache.jsonl
```
**Ожидаемо:** одна строка — запись DW-001 (текст начинается с `Subject: Cannot log in…`,
вектор — 384 измерения bge-small):
```json
{"namespace":"3f95754715e6","text":"Subject: Cannot log in after password reset\n\nI r","dims":384,"content":"{\"category\": \"Login Issues\", \"priority\":"}
```

## 7. Экономия цифрой — hit-rate из журнала

**Простыми словами:** ради этой цифры всё и затевалось: какая доля кэшируемого потока закрылась
памятью. jq фильтрует записи журнала по полю `cache` (записи `off` знаменатель не разбавляют)
и считает отношение.

```bash
make cache-stats
```
**Ожидаемо:** по трём вызовам шагов 3-5 — два попадания из трёх:
```json
{ "cached_path_calls": 3, "hits": 2, "hit_rate": 0.6666666666666666 }
```

## 8. Гейты кэша отдельно — офлайн-тесты

**Простыми словами:** всё, что демо показало живьём, закреплено тестами с фейковым эмбеддером
(рукописные векторы, $0, без ONNX): повтор → hit, парафраз → hit без кассеты, далёкий текст →
