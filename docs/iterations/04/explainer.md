# Итерация 04 — semantic cache поверх access-layer

> 🎯 Знакомство с инструментом минимальными затратами. Existence-gate, не accuracy-gate.
> Нового резюме-инструмента в этот раз нет — итерация надстраивает над access-layer (iter 3)
> **семантический кэш**: повтор или близкий по смыслу тикет отдаётся из кэша без сетевого
> вызова, а доля таких попаданий видна в журнале как hit-rate.

## Зачем это (продукт и ценность)

triagewise-lite сортирует входящие support-тикеты вымышленного SaaS *Driftwood*: по каждому тикету
модель определяет категорию, приоритет, тон, решает, нужен ли живой человек, и набрасывает черновик
ответа. Продукт при этом — не сам сортировщик, а операционный контроль над LLM-конвейером.

Реальный поток тикетов саппорта сильно повторяется: «не могу войти после сброса пароля» приходит
за день десятками формулировок, отличающихся только словами. До этой итерации каждый такой тикет
шёл в OpenAI как новый — деньги и секунды тратились на вопрос, ответ на который конвейер уже знал.
Теперь у конвейера есть память: разобрал один тикет — близкие по смыслу собратья отдаются мгновенно
и бесплатно, из кэша. Для саппорт-лида это прямое снижение счёта и латентности на самой массовой
части потока, причём управляемое: кэш включается одной переменной окружения, строгость «похожести»
— порогом, а журнал вызовов показывает, какая доля потока реально закрылась кэшем (hit-rate) — то
есть экономию видно цифрой, а не ощущением.

## 🧵 Что это дало резюме

Закрывает вторую половину north-star строки **«Productionized model routing + semantic caching»**
(первую — роутинг — дала iter 3). Артефакты-доказательства: живой прогон, где тикет-парафраз
DW-011, у которого **нет кассеты**, обслуживается семантическим попаданием (`cache=hit`,
$0.000000, строка в `logs/llm_calls.jsonl`), и `make cache-stats`, печатающий hit-rate по журналу.

## TL;DR (простыми словами)

Было — каждый вызов `route()` шёл своим путём целиком: кассета в replay, сеть в live, даже если
секунду назад разбирали почти такой же тикет. Стало — перед этим путём стоит кэш со смыслом, а не
с точным совпадением: текст тикета превращается в вектор (локальной моделью, бесплатно), и если в
сторе уже есть ответ на достаточно близкий текст — он возвращается сразу, без кассеты и без сети.
Промах идёт старым путём и по дороге пополняет стор. В SLO-журнале у каждого вызова появилось поле
`cache: hit|miss|off`, по которому `make cache-stats` считает hit-rate. По умолчанию кэш выключен —
поведение iter 3 не изменилось ни на байт.

## Что это за инструмент

Резюме-инструмента итерация не вводит (правило «один инструмент на итерацию» не потрачено), но в
зависимостях появилась библиотека-машинерия, которую стоит представить.

**fastembed** — это Python-библиотека для локального вычисления эмбеддингов: она гоняет небольшие
open-source модели через ONNX Runtime прямо в процессе, без GPU и без обращения к какому-либо API.
Здесь она решает единственную задачу — превратить текст тикета в вектор для сравнения «похожести»,
не расширяя сетевую поверхность проекта (blast radius остаётся один `acompletion`, правило 5) и не
тратя денег. Три понятия, которыми оперирует остальной документ: *эмбеддинг* — числовой вектор,
кодирующий смысл текста так, что близкие по смыслу тексты дают близкие векторы; *косинусная
близость* — число от -1 до 1, мера этой близости (у нас порог попадания 0.90); *неймспейс* —
жёсткая перегородка кэша: hash от LLM-модели, embedding-модели и системного промпта, чтобы смена
тира, промпт-версии или эмбеддера никогда не отдала ответ из чужого контекста. Конкретная модель —
`BAAI/bge-small-en-v1.5` (~30 МБ, скачивается один раз при первом живом использовании).

## Поток данных

Оператор включает кэш и разбирает тикет: `SEMANTIC_CACHE_ENABLED=1 uv run python -m app.cli.main
DW-011`. Чтобы разобрать тикет, нужен ответ модели — но прежде чем идти за ним по обычному пути,
`route()` спрашивает у кэша, не отвечали ли мы уже на достаточно похожий тикет. Для этого
`semcache.open_session()` превращает текст тикета в вектор (fastembed, локально), берёт неймспейс
(модель + эмбеддер + системный промпт) и линейно сканирует стор: косинус ≥ 0.90 в том же
неймспейсе — попадание, ответ возвращается сразу, кассета и сеть не трогаются. Промах — `route()`
идёт старой дорогой (кассета в replay, `acompletion` в record/live), а полученный свежий ответ
сессия дописывает в стор, чтобы следующий похожий тикет уже попал. Каждый исход оставляет след в
SLO-журнале — по полю `cache` таргет `make cache-stats` и считает hit-rate.

```
оператор: SEMANTIC_CACHE_ENABLED=1  python -m app.cli.main DW-011
                              │
                              ▼
                     route("cheap", messages)
                              │
                 semcache.open_session(...)
                 текст ─fastembed─▶ вектор
                              │
              скан logs/semantic_cache.jsonl
              (тот же неймспейс, cosine ≥ 0.90)
                              │
              ┌─── hit ───────┴──────── miss ───┐
              ▼                                 ▼
      ответ из кэша                обычный путь iter 0-3:
      ($0, без кассеты,          replay → кассета / live → сеть
       без сети)                                │
              │                    session.store(свежий ответ)
              │                                 │
              └────────────┬────────────────────┘
                           ▼
            slo.log_call(..., cache=hit|miss|off)
                           ▼
              logs/llm_calls.jsonl ──▶ make cache-stats (hit-rate)
```

| Инструмент | Что делает | Куда пишет |
|---|---|---|
| `semcache.open_session()` (`app/llm/semcache.py`) | эмбеддит тикет, ищет близкий ответ в неймспейсе, отдаёт `route()` готовую сессию hit/miss/off | ничего (только читает стор) |
| fastembed (`default_embedder()`) | локально считает вектор текста, модель `bge-small-en-v1.5` | ничего (модель кэшируется в `~/.cache/huggingface` при первой загрузке) |
| `Session.store()` | после промаха дописывает свежий ответ (неймспейс, текст, вектор, ответ) одной JSONL-строкой | `logs/semantic_cache.jsonl` |
| `slo.log_call()` (расширен) | пишет след вызова, теперь с полем `cache: hit\|miss\|off` | `logs/llm_calls.jsonl` |
| `make cache-stats` | jq-агрегация журнала: сколько вызовов шло кэшируемым путём, сколько попало, hit-rate | stdout |

Честные оговорки — чего в этой итерации НЕ происходит. Кэш **выключен по умолчанию**: без
`SEMANTIC_CACHE_ENABLED=1` конвейер работает байт-в-байт как в iter 3, и все записи в журнале несут
`cache=off`. В режиме `record` кэш **полностью отключается** даже включённый — иначе попадание
молча съело бы запись кассеты, от которой зависит последующий replay. Никакого вытеснения/TTL нет:
стор только растёт (попадания его не пополняют, так что повторные прогоны сходятся к стабильному
состоянию). И это кэш только для *собственных* вызовов приложения через `route()` — promptfoo и
будущий Phoenix-judge владеют своими вызовами и мимо кэша.

## Карта «где в коде»

Номера строк — ориентир на момент итерации; надёжнее искать по именам символов.

1. **Модуль кэша** — `app/llm/semcache.py` (весь файл новый). Вся механика собрана за одним швом:
   `open_session()` (`semcache.py:54`) решает судьбу вызова — при выключенном кэше, режиме
   `record` или некэшируемых сообщениях возвращает off-сессию, иначе эмбеддит текст, сканирует
   стор и отдаёт hit (с готовым контентом) или miss (с местом для записи). `route()` при этом
   ничего не знает про векторы, пороги и неймспейсы — он только секвенирует.

   ```python
   def open_session(
       settings: Settings, model: str, messages: Messages, embedder: Embedder | None = None
   ) -> Session:
       if not settings.semantic_cache_enabled or settings.llm_mode == "record":
           return Session("off")
       text = cacheable_text(messages)
       if text is None:
           return Session("off")
       embedder = embedder or default_embedder(settings.semantic_cache_embed_model)
       vector = embedder(text)
       namespace = namespace_key(model, settings.semantic_cache_embed_model, messages)
       cached = lookup(...)
       if cached is not None:
           return Session("hit", content=cached)
       return Session("miss", _miss_dest=(settings.semantic_cache_path, namespace, text, vector))
   ```

2. **Сессия как результат** — `Session` (`app/llm/semcache.py:39`). Это dataclass с исходом
   (`state`), контентом попадания и «адресом» для записи промаха; его метод `store()` — no-op для
   hit/off-сессий, поэтому `route()` зовёт его безусловно и не ветвится.

   ```python
   @dataclass
   class Session:
       state: CacheState
       content: str | None = None  # set on hit
       _miss_dest: tuple[Path, str, str, list[float]] | None = None

       def store(self, content: str) -> None:
           if self._miss_dest is None:
               return
           path, namespace, text, vector = self._miss_dest
           append_entry(path, namespace, text, vector, content)
   ```

3. **Неймспейс-перегородка** — `namespace_key()` (`app/llm/semcache.py:89`). Ключ считается как
   sha256 от LLM-модели, имени embedding-модели и всех сообщений до эмбеддируемого user-хода.
   Embedding-модель попала в ключ по итогам ревью: векторы разных эмбеддеров живут в несравнимых
   пространствах, и флип `SEMANTIC_CACHE_EMBED_MODEL` поверх живого стора обязан деградировать в
   промах, а не сравнивать (или ронять на разной размерности) чужие векторы.

4. **Стор и его живучесть** — `append_entry()` + `_load()` (`app/llm/semcache.py:121` и `:132`).
   Стор устроен как JSONL по образцу SLO-журнала: промах дописывает одну компактную строку
   (`open(..., "a")`), а чтение парсит построчно и молча пропускает битую строку — оборванная
   запись стоит одной записи, а не всего кэша.

   ```python
   def _load(path: Path) -> list[dict[str, Any]]:
       ...
       for line in path.read_text().splitlines():
           try:
               entries.append(json.loads(line))
           except json.JSONDecodeError:  # a torn line degrades to one skipped entry, not a crash
               continue
       return entries
   ```

5. **Ленивый эмбеддер** — `default_embedder()` (`app/llm/semcache.py:145`). fastembed импортируется
   внутри функции — зеркало лителлмовской дисциплины ленивости: replay-тесты, CI и путь с
   выключенным кэшем никогда не загружают ONNX. `lru_cache` держит построенную модель на процесс,
   а сам эмбеддер — просто `Callable[[str], list[float]]`, поэтому тесты подсовывают фейк с
   рукописными векторами.

   ```python
   @lru_cache(maxsize=2)
   def default_embedder(model_name: str) -> Embedder:
       from fastembed import TextEmbedding  # lazy: cache-off paths never load ONNX

       model = TextEmbedding(model_name=model_name)
       ...
   ```

6. **Интеграция в chokepoint** — `route()` (`app/llm/router.py:32`). Роутер открывает сессию до
   диспатча режимов; попадание подменяет контент, промах проваливается в старый replay/live-путь,
   и в конце — безусловные `cache.store(content)` и **единственный** `slo.log_call(...,
   cache=cache.state)`. Ранний return и второй лог-сайт из первой версии убраны полировкой.

   ```python
   cache = semcache.open_session(settings, model, messages, embedder)
   if cache.content is not None:
       content = cache.content
   elif settings.llm_mode == "replay":
       ...  # кассета, как в iter 0-3
   else:
       ...  # _live_completion, как в iter 3

   cache.store(content)  # no-op unless this call was a cache miss
   slo.log_call(..., cache=cache.state)
   ```

7. **След в журнале** — `CacheState` и `CallRecord.cache` (`app/llm/slo.py:27` и `:38`).
   Трёхзначный Literal `hit|miss|off` устроен как сосед `CostSource`; параметр `cache` в
   `log_call()` сделан обязательным, чтобы будущий вызов не смог молча залогировать `off` и
   разбавить знаменатель hit-rate.

8. **Ручки в Settings** — `app/config.py:39-46`. Четыре поля: `semantic_cache_enabled` (дефолт
   `False` — replay-путь и CI не изменились), `semantic_cache_threshold` (0.90),
   `semantic_cache_embed_model` и `semantic_cache_path` (`logs/semantic_cache.jsonl`, рядом с
   SLO-журналом в гитигноренном `logs/`). Включение кэша — env-флаг, как переключение тиров.

9. **Метрика hit-rate** — таргет `cache-stats` (`Makefile:55`). Правило 9 в действии: jq
   отфильтровывает записи кэшируемого пути (`hit|miss`), связывает счётчики через `as` и печатает
   `{cached_path_calls, hits, hit_rate}`; записи `off` знаменатель не разбавляют.

10. **Фикстура-парафраз** — DW-011 в `fixtures/tickets.jsonl`. Тот же смысл, что DW-001 («не могу
    войти после сброса пароля»), другими словами — живое доказательство, что кэш семантический:
    кассеты у DW-011 нет, а ответ есть.

11. **Тесты** — `tests/test_semcache.py` (9 тестов, всё офлайн с фейковым эмбеддером): повтор →
    hit и стор не растёт; парафраз → hit без кассеты; далёкий текст → промах и честный
    `FileNotFoundError`; чужой системный промпт → нет попадания (перегородка); флип embed-модели →
    промах, не краш; битый стор → промах, не краш; выключенный кэш → поведение iter 3; subprocess
    -проверка, что `import app.llm.router` не тянет fastembed; и чистая математика `cosine`.
